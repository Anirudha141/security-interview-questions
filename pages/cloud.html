Cloud Security

30 interview questions covering AWS, Azure, GCP, IAM, and cloud-native security

Q1 What is the Shared Responsibility Model in cloud security? Easy +

Shared Responsibility Model divides security responsibilities between cloud provider and customer based on service model.

Cloud Provider Responsibilities:

  • Physical security of data centers
  • Network infrastructure
  • Hardware maintenance
  • Hypervisor security (in IaaS)

Customer Responsibilities vary by service:

  • IaaS (EC2, VMs): OS, applications, data, network config, IAM
  • PaaS (Lambda, App Engine): Application code, data, IAM
  • SaaS (Office 365, Salesforce): Data, user access, configuration

Key point: Moving to cloud shifts some responsibilities but doesn't eliminate security needs - it changes them.

Common misunderstanding: "The cloud provider handles security" - False, significant responsibility remains with customer.

Q2 Explain AWS IAM and best practices. Medium +

AWS IAM (Identity and Access Management) controls who can do what in your AWS environment.

Key components:

  • Users: Individual identities
  • Groups: Collections of users
  • Roles: Assumable identities for services/users
  • Policies: JSON documents defining permissions

Best practices:

  • Root account: MFA, never use for daily tasks
  • Least privilege: Minimum necessary permissions
  • Use roles over access keys: For EC2, Lambda, cross-account
  • Rotate credentials: Regular key rotation
  • Use groups: Assign permissions to groups, not users
  • Enable MFA: For all users, especially privileged
  • Conditions: Restrict by IP, time, MFA
  • SCPs: Use Service Control Policies in Organizations

Policy evaluation: Explicit Deny > Explicit Allow > Implicit Deny

Q3 What is the AWS Instance Metadata Service (IMDS) and its security risks? Hard +

IMDS provides EC2 instances with metadata and temporary credentials at 169.254.169.254.

Available data:

  • Instance ID, AMI ID, hostname
  • IAM role credentials (temporary)
  • User data scripts
  • Network configuration

Security risks:

  • SSRF attacks: Force server to request metadata URL
  • Credential theft: curl http://169.254.169.254/latest/meta-data/iam/security-credentials/role-name
  • Real attack: Capital One breach (2019) exploited SSRF to steal credentials

IMDSv2 (Secure version):

  • Requires session token (PUT request first)
  • Token has TTL, prevents simple SSRF
  • Hop limit prevents proxy requests
# IMDSv2 request
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/

Best practice: Require IMDSv2, minimize IAM role permissions

Q4 How do you secure S3 buckets? Medium +

S3 Security Controls:

  • Block Public Access: Enable at account level 
    aws s3api put-public-access-block --bucket mybucket --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
  • Bucket policies: Define who can access what
  • ACLs: Legacy, prefer bucket policies
  • Encryption:
    • SSE-S3 (AWS managed keys)
    • SSE-KMS (customer managed keys)
    • Client-side encryption
  • Versioning: Protect against accidental deletion
  • MFA Delete: Require MFA for deletions
  • Access logging: Track bucket access
  • VPC Endpoints: Private access without internet

Common misconfigurations:

  • Public bucket or objects
  • Overly permissive bucket policies
  • ACL granting public access
  • Missing encryption

Q7 What are common cloud misconfigurations? Medium +

Q8 How does GCP IAM differ from AWS IAM? Medium +

GCP IAM Key Concepts:

  • Members: Google accounts, service accounts, groups, domains
  • Roles: Collection of permissions
    • Primitive (Owner, Editor, Viewer) - avoid these
    • Predefined (specific to services)
    • Custom roles
  • Resources: Organization, folders, projects, individual resources

Key differences from AWS:

GCPAWS
Hierarchical (Org > Folders > Projects)Flat accounts (with Organizations)
Roles bound at resource levelPolicies attached to identities
Service accounts = both identity and resourceSeparate IAM roles and instance profiles
IAM conditions built-inConditions in policies

GCP-specific security:

  • VPC Service Controls (data exfiltration prevention)
  • Organization policies (guardrails)
  • Workload Identity (GKE pods)
Q9 What is a CASB and why is it important? Easy +

CASB (Cloud Access Security Broker) sits between users and cloud services to enforce security policies.

Four pillars of CASB:

  • Visibility: Discover shadow IT, track cloud usage
  • Compliance: Ensure regulatory compliance (GDPR, HIPAA)
  • Data Security: DLP, encryption, tokenization
  • Threat Protection: Malware detection, anomaly detection

Deployment modes:

  • API-based: Connect to SaaS via API (post-upload)
  • Proxy-based: Inline inspection (real-time)
  • Log-based: Analyze firewall/proxy logs

Use cases:

  • Discover unsanctioned cloud apps
  • Prevent sensitive data upload to personal accounts
  • Detect compromised accounts
  • Enforce encryption on cloud storage

Vendors: Microsoft Defender for Cloud Apps, Netskope, Zscaler, Palo Alto Prisma

Q10 How do you secure serverless functions (Lambda, Azure Functions)? Hard +

Serverless Security Challenges:

  • Increased attack surface (more functions, more entry points)
  • Traditional security tools don't work
  • Event-driven, ephemeral nature
  • Third-party dependencies

Security best practices:

  • Least privilege IAM: Minimal permissions per function
  • Input validation: All event sources (API, S3, SNS, etc.)
  • Secrets management: Use Secrets Manager/Parameter Store, not env vars
  • Dependencies: Audit packages, use minimal dependencies
  • VPC: Run in VPC if accessing private resources
  • Function URLs: Secure with IAM auth if using
  • Timeouts: Set appropriate limits
  • Logging: Enable CloudWatch/equivalent

Serverless-specific vulnerabilities:

  • Event injection
  • Broken authentication on API triggers
  • Over-privileged function roles
  • Insecure secrets storage

Q13 What is AWS Organizations and Service Control Policies? Medium +

AWS Organizations provides centralized management of multiple AWS accounts.

Structure:

  • Management account (root)
  • Organizational Units (OUs)
  • Member accounts

Service Control Policies (SCPs):

  • Guardrails that limit permissions across accounts
  • Do NOT grant permissions (only restrict)
  • Apply to OUs or accounts
  • Effective permissions = SCP âˆ© IAM policy

Common SCP patterns:

// Deny leaving organization
{
  "Effect": "Deny",
  "Action": "organizations:LeaveOrganization",
  "Resource": "*"
}

// Deny disabling CloudTrail
{
  "Effect": "Deny",
  "Action": "cloudtrail:StopLogging",
  "Resource": "*"
}

// Region restriction
{
  "Effect": "Deny",
  "NotAction": ["iam:*", "sts:*"],
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {"aws:RequestedRegion": "us-east-1"}
  }
}
Q14 How do you detect compromised cloud credentials? Hard +

Detection Methods:

  • CloudTrail analysis:
    • Unusual API calls
    • Access from new IPs/regions
    • High-volume requests
    • Reconnaissance APIs (List*, Describe*)
  • GuardDuty findings:
    • Compromised credentials indicators
    • Unusual resource access
    • Crypto mining detection
  • Indicators of compromise:
    • New IAM users/access keys created
    • S3 bucket policy changes
    • Security group modifications
    • EC2 instances in unusual regions
    • Lambda functions created

Response actions:

  1. Disable compromised credentials
  2. Revoke active sessions
  3. Review CloudTrail for scope of access
  4. Check for persistence (new users, roles, keys)
  5. Scan for created resources
Q15 What is cloud security posture management (CSPM)? Easy +

CSPM continuously monitors cloud infrastructure for misconfigurations and compliance violations.

Key capabilities:

  • Asset discovery: Inventory all cloud resources
  • Configuration assessment: Check against security benchmarks
  • Compliance monitoring: CIS, NIST, PCI-DSS, HIPAA
  • Risk prioritization: Rank findings by severity
  • Remediation: Automated or guided fixes

What CSPM detects:

  • Public S3 buckets
  • Unencrypted databases
  • Overly permissive security groups
  • Missing logging
  • IAM misconfigurations

Tools:

  • AWS: Security Hub, Config
  • Azure: Defender for Cloud
  • GCP: Security Command Center
  • Third-party: Prisma Cloud, Wiz, Orca, Lacework

Q17 What is a VPC and how do you secure it? Medium +

VPC (Virtual Private Cloud) is a logically isolated network in the cloud.

VPC components:

  • Subnets: Public (internet access) and private
  • Route tables: Traffic routing rules
  • Internet Gateway: Public internet access
  • NAT Gateway: Outbound internet for private subnets
  • Security Groups: Instance-level stateful firewall
  • NACLs: Subnet-level stateless firewall

VPC security best practices:

  • Use private subnets for databases, internal services
  • Restrictive security groups (no 0.0.0.0/0 inbound)
  • VPC Flow Logs enabled
  • VPC endpoints for AWS services (avoid internet)
  • Network segmentation by environment/function
  • No default VPC usage
  • DNS logging

VPC Peering/Transit Gateway: Secure cross-VPC communication

Q18 How do you handle secrets in cloud environments? Medium +

Secrets Management Services:

  • AWS: Secrets Manager, Parameter Store
  • Azure: Key Vault
  • GCP: Secret Manager
  • HashiCorp Vault: Multi-cloud option

Best practices:

  • Never hardcode: No secrets in code, configs, or env vars
  • Rotation: Automatic secret rotation
  • Least privilege: Minimal access to secrets
  • Audit logging: Track all secret access
  • Encryption: Encrypted at rest with CMK
  • Short-lived credentials: Use dynamic secrets when possible

Application patterns:

  • Fetch at runtime (SDK/API call)
  • Inject via sidecar (Kubernetes)
  • Environment variable injection (controlled)

Scanning for exposed secrets: git-secrets, trufflehog, detect-secrets

Q19 What is cloud workload protection (CWPP)? Medium +

CWPP (Cloud Workload Protection Platform) protects workloads (VMs, containers, serverless) across cloud environments.

Key capabilities:

  • Vulnerability management: OS and application CVEs
  • Runtime protection: Behavior monitoring, threat detection
  • Container security: Image scanning, runtime protection
  • File integrity monitoring: Detect unauthorized changes
  • Network security: Microsegmentation, traffic analysis
  • Compliance: Configuration assessment

CWPP vs CSPM:

  • CSPM: Configuration and posture (control plane)
  • CWPP: Workload protection (data plane)

CNAPP = CSPM + CWPP: Cloud-Native Application Protection Platform

Vendors: Palo Alto Prisma Cloud, CrowdStrike, Wiz, Aqua Security, Sysdig

Q21 What is Azure Conditional Access? Medium +

Conditional Access provides identity-driven access control based on signals like user, device, location, and risk.

Policy components:

  • Assignments (Who):
    • Users and groups
    • Cloud apps
    • Conditions (location, device, risk level)
  • Access controls (What):
    • Block access
    • Grant access with conditions
    • Require MFA
    • Require compliant device
    • Require approved app

Common policies:

  • Require MFA for all users
  • Block legacy authentication
  • Require compliant devices for sensitive apps
  • Block access from risky locations
  • Require MFA for risky sign-ins

Integration: Works with Intune for device compliance, Identity Protection for risk signals

Q22 How do you investigate cloud security incidents? Hard +

Cloud Incident Investigation Steps:

  1. Identification:
    • Alert from GuardDuty/Defender/SCC
    • Unusual activity in logs
    • External notification
  2. Containment:
    • Disable compromised credentials
    • Isolate affected resources (security groups)
    • Revoke sessions
  3. Investigation:
    • CloudTrail/Activity Log analysis
    • Identify initial access
    • Map attacker actions
    • Identify created resources/persistence
  4. Eradication:
    • Remove unauthorized resources
    • Remove persistence mechanisms
    • Rotate all exposed credentials
  5. Recovery: Restore from known-good state
  6. Lessons learned: Improve detection and prevention

Key log sources: CloudTrail, VPC Flow Logs, DNS logs, S3 access logs, GuardDuty findings

Q23 What is cloud penetration testing scope and rules? Medium +

Cloud Provider Policies:

AWS:

  • No pre-approval needed for most services
  • Prohibited: DDoS, DNS zone walking, port flooding
  • Must own/have permission for target resources

Azure:

  • No pre-approval for owned resources
  • Follow rules of engagement
  • No testing against other tenants

GCP:

  • Permitted on your own projects
  • No notification required
  • Follow Acceptable Use Policy

Scope considerations:

  • Only test resources you own/authorized
  • Exclude shared infrastructure
  • Document scope clearly
  • Consider blast radius of actions
  • Time-bound testing windows

What to test: IAM, storage, compute configs, network security, API security, logging effectiveness

Q24 How do you implement data encryption in the cloud? Medium +

Encryption at Rest:

  • Server-side encryption:
    • Provider-managed keys (SSE-S3, Azure Storage encryption)
    • Customer-managed keys (SSE-KMS, CMK)
    • Customer-provided keys (SSE-C)
  • Client-side encryption: Encrypt before upload

Key Management:

  • AWS KMS: Managed key service
  • Azure Key Vault: Keys, secrets, certificates
  • GCP Cloud KMS: Managed encryption keys
  • HSM options: CloudHSM, Dedicated HSM

Encryption in Transit:

  • TLS 1.2+ for all connections
  • Certificate management (ACM, Let's Encrypt)
  • VPC endpoints (private connectivity)

Best practices:

  • Enable encryption by default
  • Use CMK for sensitive data
  • Rotate keys regularly
  • Separate keys by environment/classification
Q25 What are privilege escalation paths in cloud environments? Hard +

AWS Privilege Escalation:

  • IAM: iam:CreatePolicyVersion, iam:AttachUserPolicy, iam:PutUserPolicy
  • PassRole + Service: Lambda, EC2, Glue with overprivileged roles
  • AssumeRole: Assume more privileged roles
  • Instance metadata: Steal role credentials from EC2
  • CloudFormation: Deploy with PassRole to privileged role

Azure Privilege Escalation:

  • Service Principal credential reset
  • Role assignment (User Access Administrator)
  • Managed Identity abuse
  • Key Vault access

GCP Privilege Escalation:

  • setIamPolicy on resources
  • Service account key creation
  • ActAs permission
  • Deployment Manager with privileged SA

Tools: Pacu (AWS), PMapper, CloudGoat, AzureHound

Q26 How do you secure multi-cloud environments? Hard +

Multi-Cloud Security Challenges:

  • Different security models per provider
  • Inconsistent IAM/RBAC
  • Multiple attack surfaces
  • Complex visibility

Strategies:

  • Unified identity:
    • Centralized IdP (Okta, Azure AD)
    • Federation to all cloud providers
    • Consistent MFA policies
  • Centralized visibility:
    • Multi-cloud SIEM
    • CSPM spanning all providers
    • Unified logging
  • Consistent policies:
    • Policy-as-code (OPA, Sentinel)
    • Unified compliance frameworks
    • Cross-cloud security standards
  • Network security:
    • Secure cross-cloud connectivity
    • Consistent network controls

Tools: Multi-cloud CSPM (Wiz, Prisma Cloud), Terraform for IaC

Q27 What is AWS GuardDuty and how does it work? Medium +

AWS GuardDuty is a managed threat detection service that continuously monitors for malicious activity.

Data sources:

  • CloudTrail management events
  • CloudTrail S3 data events
  • VPC Flow Logs
  • DNS logs
  • EKS audit logs
  • RDS login activity
  • Lambda network activity

Finding types:

  • Recon: Port scanning, unusual API calls
  • Instance compromise: Crypto mining, C2 communication
  • Account compromise: Credential exfiltration indicators
  • Bucket compromise: Unusual S3 access patterns

Response automation:

  • EventBridge rules trigger on findings
  • Lambda functions for automated response
  • Security Hub integration

Setup: Enable with one click, no agents needed

Q28 How do you secure cloud databases? Medium +

Cloud Database Security:

  • Network isolation:
    • Private subnets only
    • No public accessibility
    • Security groups restrictive
    • VPC endpoints for access
  • Authentication:
    • IAM authentication when available
    • Strong passwords
    • Disable default accounts
  • Encryption:
    • Encryption at rest (mandatory)
    • TLS for connections
    • Customer-managed keys for sensitive data
  • Backup security:
    • Encrypted backups
    • Separate backup account
    • Test restoration
  • Monitoring:
    • Audit logging enabled
    • Slow query logging
    • Connection monitoring

Services: RDS, Aurora (AWS), Azure SQL, Cloud SQL (GCP)
