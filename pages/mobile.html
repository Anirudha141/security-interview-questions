Mobile Application Security

30 interview questions covering Android, iOS, reverse engineering, and app security

← Back to Categories
Q1 What is the OWASP Mobile Top 10? Easy +

OWASP Mobile Top 10 (2024):

  1. M1: Improper Credential Usage - Hardcoded credentials, improper storage
  2. M2: Inadequate Supply Chain Security - Third-party libraries, SDKs
  3. M3: Insecure Authentication/Authorization - Weak auth mechanisms
  4. M4: Insufficient Input/Output Validation - Injection, XSS in WebViews
  5. M5: Insecure Communication - Cleartext traffic, weak TLS
  6. M6: Inadequate Privacy Controls - PII exposure, tracking
  7. M7: Insufficient Binary Protections - Reverse engineering, tampering
  8. M8: Security Misconfiguration - Debug flags, backup enabled
  9. M9: Insecure Data Storage - Sensitive data in logs, shared prefs
  10. M10: Insufficient Cryptography - Weak algorithms, poor key management

Related: OWASP MASTG (Mobile Application Security Testing Guide), MASVS (Verification Standard)

Q2 How do you set up a mobile app testing environment? Medium +

Android Environment:

  • Device/Emulator: Rooted device or emulator (Genymotion, Android Studio)
  • Proxy: Burp Suite with certificate installed
  • Tools: adb, Frida, objection, apktool, jadx, Drozer
  • Root tools: Magisk (recommended) or SuperSU

iOS Environment:

  • Device: Jailbroken iPhone (checkra1n, unc0ver)
  • Proxy: Burp Suite with profile installed
  • Tools: Frida, objection, Cycript, class-dump, Hopper/IDA
  • SSH: OpenSSH on device for file access

Certificate installation challenges:

  • Android 7+: User certs not trusted by default
  • Solutions: Magisk module, patch APK, Frida SSL unpinning
  • iOS: Install profile + trust in settings
Q3 Explain SSL/TLS pinning and how to bypass it. Hard +

SSL Pinning restricts which certificates an app accepts, preventing MITM attacks even with a trusted CA certificate installed.

Types:

  • Certificate pinning: Pin entire certificate
  • Public key pinning: Pin public key (survives cert renewal)
  • Hash pinning: Pin hash of cert/key

Bypass techniques:

  • Frida scripts: 
    frida -U -f com.app -l ssl-unpin.js
  • objection: 
    objection -g com.app explore
android sslpinning disable
  • Patch APK: Modify smali code, repackage, sign
  • Xposed/Magisk modules: SSLUnpinning, TrustMeAlready
  • Network config: Modify network_security_config.xml

iOS: SSL Kill Switch, Frida scripts, binary patching

Q4 What is the Android application sandbox? Easy +

Android Sandbox isolates apps from each other and the system using Linux kernel features.

Key components:

  • UID isolation: Each app gets unique Linux UID
  • Process isolation: Apps run in separate processes
  • File system isolation: App-specific directories
  • SELinux: Mandatory access control policies

App data locations:

  • /data/data/[package] - Private app storage
  • /data/app/[package] - APK installation
  • /sdcard/Android/data/[package] - External storage

Security implications:

  • Apps cannot read other apps' private data (without root)
  • IPC requires explicit permissions
  • System resources need runtime permissions
Q5 How do you decompile and analyze an Android APK? Medium +

APK Analysis Workflow:

  1. Extract APK contents: 
    unzip app.apk -d extracted/
  2. Decompile to Smali: 
    apktool d app.apk -o output/
  3. Decompile to Java: 
    jadx app.apk -d jadx-output/
# Or use jadx-gui for visual analysis
  4. Analyze:
    • AndroidManifest.xml - Permissions, components, configs
    • strings.xml - Hardcoded strings
    • Source code - Business logic, API endpoints
    • lib/ - Native libraries (.so files)
    • assets/ - Certificates, databases, configs

Tools: jadx, apktool, dex2jar, JD-GUI, Bytecode Viewer, MobSF

Q6 What is Frida and how do you use it for mobile testing? Hard +

Frida is a dynamic instrumentation toolkit that injects JavaScript into running processes to hook functions, trace calls, and modify behavior.

Setup:

# Install Frida
pip install frida-tools

# Push frida-server to device
adb push frida-server /data/local/tmp/
adb shell chmod +x /data/local/tmp/frida-server
adb shell /data/local/tmp/frida-server &

Basic usage:

# List processes
frida-ps -U

# Attach to app
frida -U com.target.app

# Run script
frida -U -f com.target.app -l script.js

Example script (hook function):

Java.perform(function() {
    var Activity = Java.use("com.app.LoginActivity");
    Activity.checkPassword.implementation = function(pass) {
        console.log("Password: " + pass);
        return true; // Always return true
    };
});

Use cases: SSL unpinning, root detection bypass, function tracing, runtime modification

Q7 How does iOS app signing work? Medium +

iOS Code Signing Components:

  • Certificate: Developer/distribution certificate from Apple
  • Provisioning Profile: Links app ID, device UDIDs, entitlements
  • Entitlements: Capabilities (keychain access, push notifications)

Signing types:

  • Development: Limited devices, debugging enabled
  • Ad Hoc: Up to 100 devices, no App Store
  • Enterprise: Unlimited devices within organization
  • App Store: Apple re-signs for distribution

Security implications:

  • Cannot run unsigned code (without jailbreak)
  • Tampering invalidates signature
  • Re-signing requires valid certificate

For testing: Re-sign with your cert using ios-deploy, iReSign, or manual codesign

Q8 What are Android content providers and their security risks? Medium +

Content Providers allow apps to share structured data with other apps through a standardized interface (content:// URIs).

Security risks:

  • Exported providers: android:exported="true" allows any app access
  • SQL injection: Improper input sanitization in queries
  • Path traversal: In file-based providers
  • Information disclosure: Sensitive data exposed
  • Permission bypass: Weak permission requirements

Testing with Drozer:

run app.provider.info -a com.target.app
run app.provider.query content://com.app/users
run app.provider.query content://com.app/users --projection "* FROM users;--"

Mitigations:

  • Set android:exported="false" when not needed
  • Use signature-level permissions
  • Parameterized queries
  • Input validation

Common detection methods:

  • Check for su binary, Superuser.apk
  • Check for Magisk, Cydia, Sileo
  • Test directory write permissions
  • Check system properties (ro.build.tags)
  • Check for hook frameworks (Frida, Xposed)

Q10 What is the iOS Keychain and how do you assess its security? Medium +

iOS Keychain is a secure storage container for sensitive data like passwords, tokens, and certificates.

Security classes:

  • kSecAttrAccessibleWhenUnlocked - Available when device unlocked
  • kSecAttrAccessibleAfterFirstUnlock - Available after first unlock (common)
  • kSecAttrAccessibleAlways - Always available (deprecated, insecure)
  • kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly - Most secure

Testing (jailbroken):

# Using keychain-dumper
./keychain-dumper

# Using objection
objection -g com.app explore
ios keychain dump

Common issues:

  • Using AccessibleAlways for sensitive data
  • Not using ThisDeviceOnly variants
  • Storing plaintext secrets
  • Not clearing keychain on logout/uninstall
Q11 What are WebView vulnerabilities in mobile apps? Medium +

WebView embeds web content in native apps, creating a bridge between web and native code.

Common vulnerabilities:

  • JavaScript enabled: setJavaScriptEnabled(true) enables XSS
  • JavaScript Interface: addJavascriptInterface() can expose Java methods 
    // Android < 4.2: Call any public method
webView.addJavascriptInterface(new Object(), "Android");
  • File access: setAllowFileAccess(true) enables file:// URLs
  • Universal file access: Read any file via XSS
  • Insecure URL loading: Loading attacker-controlled URLs
  • SSL errors ignored: onReceivedSslError calling proceed()

Testing:

  • Inject JavaScript through user input
  • Check WebView settings in decompiled code
  • Test deep links that open WebViews
Q12 How do you test for insecure data storage on mobile? Medium +

Android locations to check:

  • /data/data/[package]/shared_prefs/ - SharedPreferences XML
  • /data/data/[package]/databases/ - SQLite databases
  • /data/data/[package]/files/ - Internal storage
  • /sdcard/ - External storage (world-readable)
  • Application logs (adb logcat)
  • Backup files

iOS locations to check:

  • /var/mobile/Containers/Data/Application/[UUID]/
  • Documents/, Library/, tmp/
  • NSUserDefaults plist files
  • Core Data / SQLite databases
  • Keychain
  • Cookies, cache

What to look for:

  • Plaintext credentials, tokens, PII
  • Unencrypted databases
  • Sensitive data in logs
  • Backup inclusion of sensitive files
Q13 What are Android intents and their security implications? Medium +

Intents are messaging objects for inter-component and inter-app communication.

Types:

  • Explicit: Target specific component
  • Implicit: Declare action, system resolves receiver

Security issues:

  • Intent interception: Malicious app registers for same intent filter
  • Intent spoofing: Send crafted intents to exported components
  • Pending Intent hijacking: Mutable PendingIntents can be modified
  • Broadcast sniffing: Unprotected broadcasts readable by other apps
  • Data exposure: Sensitive data in intent extras

Testing with adb:

# Start activity
adb shell am start -n com.app/.DeepLinkActivity -d "scheme://path"

# Send broadcast
adb shell am broadcast -a com.app.ACTION

Mitigations: Use explicit intents, set permissions, use LocalBroadcastManager

Q14 How do you perform static analysis of iOS apps? Medium +

Getting the IPA:

  • iTunes (older versions)
  • iMazing, Apple Configurator
  • Frida ios-dump (from jailbroken device)
  • Clutch (decrypt App Store apps)

Analysis steps:

  1. Extract IPA: 
    unzip app.ipa -d extracted/
  2. Analyze Info.plist: URL schemes, permissions, configurations
  3. Check binary: 
    # Check protections
otool -hv Payload/App.app/App

  4. Disassemble: Hopper, IDA Pro, Ghidra
  5. Class dump: 
    class-dump Payload/App.app/App > headers.h
  6. String analysis: Look for URLs, keys, secrets

Tools: MobSF, Hopper, IDA, class-dump, otool

Q16 How do you test deep links for vulnerabilities? Medium +

Deep links allow external navigation directly to app content via custom URL schemes or App Links/Universal Links.

Find deep links:

  • AndroidManifest.xml: <intent-filter> with <data>
  • iOS Info.plist: CFBundleURLSchemes, Associated Domains

Vulnerabilities to test:

  • Open redirect: app://redirect?url=http://evil.com
  • XSS via WebView: app://webview?url=javascript:alert(1)
  • Authentication bypass: Access authenticated screens
  • Parameter injection: Modify sensitive parameters
  • IDOR: Access other users' resources

Testing:

# Android
adb shell am start -a android.intent.action.VIEW -d "scheme://path?param=value"

# iOS (simulator)
xcrun simctl openurl booted "scheme://path"
Q17 What binary protections should mobile apps implement? Medium +

Android protections:

  • Code obfuscation: ProGuard, R8, DexGuard
  • Native code: Critical logic in .so files
  • String encryption: Encrypt sensitive strings
  • Integrity checks: Verify APK signature at runtime
  • Debugger detection: Check for debugging flags
  • Emulator detection: Detect non-real devices

iOS protections:

  • PIE (Position Independent Executable): ASLR support
  • ARC (Automatic Reference Counting): Memory safety
  • Stack canaries: Buffer overflow protection
  • Code obfuscation: LLVM-based obfuscators
  • Anti-debugging: ptrace detection, sysctl checks

Check protections:

# iOS
otool -hv binary | grep PIE

# Android - check for obfuscation
jadx app.apk  # Look for a/b/c class names
Q18 How do you intercept traffic from apps that don't respect system proxy? Hard +

Apps may bypass proxy by:

  • Using custom HTTP libraries that ignore proxy settings
  • Hardcoding connections
  • Using non-HTTP protocols

Solutions:

  • VPN-based interception:
    • Use ProxyDroid (Android, requires root)
    • Run transparent proxy on network level
  • iptables (rooted Android): 
    iptables -t nat -A OUTPUT -p tcp --dport 443 -j DNAT --to-destination <proxy-ip>:8080
  • Frida hook: Hook HTTP library to add proxy 
    // Hook OkHttp client builder to add proxy
  • Patch APK: Modify code to use proxy
  • Network level: Run proxy on router/gateway

For non-HTTP: Use Wireshark, tcpdump, or protocol-specific tools

Q19 What is the Android Backup vulnerability? Easy +

Android Backup allows extracting app data via adb if android:allowBackup="true" (default).

Exploitation:

# Create backup (requires user confirmation on device)
adb backup -f backup.ab com.target.app

# Extract backup
java -jar abe.jar unpack backup.ab backup.tar
tar -xvf backup.tar

What you can extract:

  • SharedPreferences (may contain tokens, credentials)
  • SQLite databases
  • Internal files
  • Cache data

Security implications:

  • Physical access to device = data extraction
  • Malware can trigger backup and steal data
  • Data persists after logout

Mitigation:

<application android:allowBackup="false">

Or use android:fullBackupContent to exclude sensitive files.

Q20 How do you analyze native libraries (.so files) in Android apps? Hard +

Native libraries contain compiled C/C++ code, often used for performance-critical or security-sensitive operations.

Analysis workflow:

  1. Extract: From APK's lib/ directory (arm64-v8a, armeabi-v7a, x86)
  2. Identify architecture: 
    file libapp.so
  3. String analysis: 
    strings libapp.so | grep -i password
rabin2 -z libapp.so
  4. Disassemble:
    • IDA Pro (industry standard)
    • Ghidra (free, powerful)
    • radare2/rizin
  5. Find JNI functions: Look for Java_com_package_Class_method

Dynamic analysis with Frida:

Interceptor.attach(Module.findExportByName("libapp.so", "encrypt"), {
    onEnter: function(args) {
        console.log("Input: " + Memory.readUtf8String(args[0]));
    }
});
Q21 What are iOS URL schemes and their security risks? Medium +

URL Schemes allow apps to be launched via custom URLs (e.g., myapp://action).

Security risks:

  • Scheme hijacking: Malicious app registers same scheme, intercepts calls
  • Sensitive data in URL: Tokens, credentials in scheme parameters
  • Action triggering: Force app to perform actions (payment, data deletion)
  • XSS in WebView: Scheme opens WebView with attacker URL

Finding schemes (Info.plist):

<key>CFBundleURLTypes</key>
<array>
    <dict>
        <key>CFBundleURLSchemes</key>
        <array>
            <string>myapp</string>
        </array>
    </dict>
</array>

Testing:

# Safari
myapp://action?param=value

# Command line
xcrun simctl openurl booted "myapp://test"

Mitigation: Use Universal Links (verified domain ownership), validate all inputs

Q23 What is tapjacking and how do you test for it? Medium +

Tapjacking is the mobile equivalent of clickjacking - overlaying malicious UI to trick users into tapping hidden elements.

Android vulnerability:

  • Apps with android:filterTouchesWhenObscured="false" (default)
  • Attacker creates overlay using SYSTEM_ALERT_WINDOW permission
  • User sees attacker UI, taps on hidden victim app buttons

Testing:

  1. Check if app sets filterTouchesWhenObscured="true"
  2. Create PoC overlay app
  3. Test if critical actions can be triggered through overlay

Android 12+ changes: Untrusted touch events are blocked by default

Mitigation:

// In XML
android:filterTouchesWhenObscured="true"

// In code
view.setFilterTouchesWhenObscured(true);
Q24 How do you test for clipboard vulnerabilities? Easy +

Clipboard risks:

  • Sensitive data copied to clipboard (passwords, tokens)
  • Other apps can read clipboard contents
  • Clipboard history may persist data
  • Screenshots may capture clipboard indicators

Testing approach:

  1. Copy sensitive data in app (password, credit card)
  2. Check if app clears clipboard after use
  3. Check clipboard from another app or adb: 
    adb shell service call clipboard 2 s16 com.app
  4. Verify clipboard clearing on app background/close

Android 10+ restrictions: Only foreground apps can read clipboard

iOS 14+ indicators: Banner shown when apps read clipboard

Best practices:

  • Don't allow copying sensitive data
  • Clear clipboard after paste
  • Use secure text fields that prevent copy

Q26 How do you test biometric authentication implementation? Hard +

Biometric testing areas:

  • Fallback mechanisms: Can biometric be bypassed with PIN/password?
  • Crypto binding: Is biometric tied to cryptographic key?
  • Event-based vs crypto-based: Event-based is weaker
  • Lockout policy: Multiple failed attempts handling

Android testing:

// Hook BiometricPrompt result
Java.use("androidx.biometric.BiometricPrompt$AuthenticationCallback")
  .onAuthenticationSucceeded.implementation = function() {
    // Bypass
}

iOS testing:

// Hook LAContext
var LAContext = ObjC.classes.LAContext;
Interceptor.attach(LAContext["- evaluatePolicy:localizedReason:reply:"].implementation, {
    // Modify result
});

Key questions:

  • Does server know about biometric auth or just client?
  • Can the auth result callback be hooked?
  • Is sensitive data protected by biometric-bound keys?
Q27 What are common cryptographic mistakes in mobile apps? Medium +

Common cryptographic mistakes:

  • Hardcoded keys: Encryption keys in source code or assets
  • Weak algorithms: MD5, SHA1, DES, RC4
  • ECB mode: Reveals patterns in encrypted data
  • Static IV: Using same IV for all encryption
  • Insufficient key length: RSA < 2048 bits, AES < 128 bits
  • Custom crypto: Rolling own implementation
  • No integrity check: Encryption without MAC (use GCM or encrypt-then-MAC)
  • Predictable random: Using Math.random() for crypto
  • Key derivation: Using password directly as key (use PBKDF2, scrypt)

What to look for in code:

// Bad patterns
Cipher.getInstance("AES/ECB/PKCS5Padding")
SecretKeySpec key = new SecretKeySpec(hardcoded, "AES")
MessageDigest.getInstance("MD5")

Q29 What is the difference between MASVS L1 and L2? Easy +

MASVS (Mobile Application Security Verification Standard) defines security requirements at different levels.

MASVS-L1 (Standard Security):

  • Baseline security for all mobile apps
  • Focus on common vulnerabilities
  • Appropriate for apps without sensitive data
  • Requirements: Secure storage, network security, authentication basics

MASVS-L2 (Defense-in-Depth):

  • For apps handling sensitive data
  • All L1 requirements plus additional controls
  • Binary protections against reverse engineering
  • Anti-tampering measures
  • Runtime protection

MASVS-R (Resilience Against Reverse Engineering):

  • Additional requirements focused on client-side protection
  • Can be added to L1 or L2
  • Obfuscation, anti-debugging, integrity checks
